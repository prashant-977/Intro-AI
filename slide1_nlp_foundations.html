Before a computer can understand what a sentence means, it first has to understand what it is. Upstream tasks are the foundational processing steps that happen behind the scenes — the grammar and vocabulary layer that every NLP application depends on, even though users never see them.

Tokenisation

Splitting raw text into individual units — words, subwords, or sentences — so the computer has something to work with.

"I love NLP!" → ["I", "love", "NLP", "!"]

Part-of-speech tagging

Labelling each token as a noun, verb, adjective, etc. Tells the model the grammatical role of every word before meaning is inferred.

Word embeddings

Converting words into numbers (vectors) that capture semantic similarity. "King" and "Queen" end up numerically close. Text becomes maths.

The analogy that works in a classroom

Upstream tasks are like teaching a child to read, spell, and understand grammar — before you ever ask them to write an essay, give a speech, or read a contract. The foundational skills are invisible in the finished product, but nothing works without them.

Downstream tasks are the practical, user-facing applications that make NLP useful in the real world. They take the structured output of upstream processing and turn it into something a manager can actually act on. If upstream tasks are the building blocks, downstream tasks are the finished products — the applications that directly solve user problems.

Every notebook in the opendata-education/TextAnalytics repo is a downstream task. The upstream layer is already handled for you.
For decades, each downstream task required its own custom-built system. The shift to large language models changed that: a single model, trained once at enormous scale, can now perform almost any downstream task simply by being given the right instructions in plain English. That shift is what makes this moment in NLP genuinely different from everything that came before.
1
1980s – 2000s
Rule-based systems
Engineers hand-wrote explicit rules: "if the email contains 'free money', label it spam." Worked for narrow, stable domains. Broke immediately when language evolved or context changed. Required constant human maintenance.
2
2000s – 2017
Statistical machine learning
Models learned patterns from labelled data rather than rules. Each downstream task still needed its own model trained from scratch. Powerful but expensive — large datasets required for every new application.
3
2017 – 2022
Pre-trained transformers (BERT, GPT-2, T5)
Models pre-trained on billions of words of internet text, then fine-tuned on small task-specific datasets. Transfer learning meant that a model trained on general language could be adapted to legal, medical, or financial text with far less data. The upstream/downstream distinction became explicit in model architecture.
4
2022 – present
Large language models — one model, all tasks
ChatGPT, Claude, Gemini, and Copilot collapsed the distinction between tasks. You no longer deploy a sentiment model, a summarisation model, and a classification model separately. You write a prompt — and the same model does all three. The bottleneck shifted from building the model to instructing it well.

What stayed the same

The downstream tasks themselves — sentiment analysis, NER, summarisation, classification — did not change. LLMs just became a faster, more flexible way to perform them. Understanding what a downstream task is still matters, because it tells you what to ask for.

What changed for business users

The barrier moved from can we build this? to can we instruct this well? Prompt engineering — writing clear, specific, structured instructions — is now the primary skill separating useful AI output from generic noise.

Opening question for the room

Think of one task at work or university that involves reading a lot of text and pulling out specific information. Which downstream task category does that fall into — classification, extraction, summarisation, or generation? And could you describe that task clearly enough for an AI to do it for you?
